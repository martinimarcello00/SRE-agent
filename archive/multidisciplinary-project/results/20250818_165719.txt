  - panic stack: "no reachable servers" coming from main.initializeDatabase → container terminates.
- Kubernetes service list shows mongodb-geo service exposing port 27017 (ClusterIP 10.96.90.223:27017).
- mongodb-geo pod logs show MongoDB started and listening on port 27017 (multiple "Listening on ... port:27017" messages) and the init script completed successfully.
- Pod status/describe for geo: container restarted 3 times, State Terminated Reason=Error, BackOff events.

Reasoning / troubleshooting steps I ran
1. Listed all pods across namespaces to find failing pods; geo was in Error/CrashLoopBackOff.
2. Described the failing geo pod to see restart counts, image, env, and events.
3. Pulled the geo container logs — saw the DB URL the app attempted to use and the panic "no reachable servers".
4. Listed services in the test-hotel-reservation namespace — confirmed mongodb-geo service port is 27017.
5. Inspected the mongodb-geo pod logs — verified MongoDB is up and listening on 27017 and initialization completed.
6. Correlated the two: geo tries to reach port 27777 while MongoDB listens on 27017 → connection attempts fail → app panics.

Impact
- The geo microservice is not available (restarting/crashing), which will break any functionality depending on geo (likely search/recommendation flows that need geolocation lookups). The pod is in a crash loop, consuming restarts and causing degraded service.

Likely causes / where the misconfiguration lives
- Misconfigured database URL / port for the geo service (environment variable or config file) set to mongodb-geo:27777 instead of mongodb-geo:27017.
- Less likely: service port was intended to be remapped but the mongodb service was created with the wrong port. (Either side could be fixed; logs point to geo using 27777.)

Recommended (non-action) remediation options
- Correct the geo service configuration so it uses mongodb-geo:27017 (or update the mongodb service to expose 27777 if that is the intended contract).
- Redeploy geo with the corrected DB URL; ensure health checks and graceful handling for DB connection failures (avoid panic on init to allow better restart behaviour).
